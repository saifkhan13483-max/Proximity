2.1 — Railway entry point
   • Create a railway.toml in the backend/ folder:
       [build]
       builder = "nixpacks"
       [deploy]
       startCommand = "node server.js"
       healthcheckPath = "/health"
       healthcheckTimeout = 300
       restartPolicyType = "on_failure"

2.2 — Dynamic port binding
   • In server.js, replace any hardcoded PORT = 3001 with:
       const PORT = process.env.PORT || 3001;
   • Railway dynamically assigns the port — this is mandatory.

2.3 — CORS configuration (security fix)
   • Remove "origin: true" (this allows ALL origins — insecure)
   • Replace with an explicit allowlist sourced from the ALLOWED_ORIGINS
     environment variable (comma-separated string), e.g.:
       const allowedOrigins = process.env.ALLOWED_ORIGINS?.split(",")
         .map(o => o.trim()) || ["http://localhost:5173"];
   • The CORS middleware must reject requests from unlisted origins.

2.4 — Reverse-proxy trust
   • Add this line BEFORE any route handlers in server.js:
       app.set("trust proxy", 1);
   • Required for Railway's reverse-proxy so req.ip, req.protocol,
     and secure cookies work correctly.

2.5 — Health check endpoint
   • Add a GET /health route returning:
       {
         "status": "ok",
         "uptime": process.uptime(),
         "timestamp": new Date().toISOString()
       }
   • Response status: 200

2.6 — CRITICAL: Fix ephemeral filesystem data loss
   • Railway's filesystem resets on every deploy. Any data stored in
     users.json or contacts.json WILL BE WIPED.
   • REQUIRED FIX (choose one):
       Option A (recommended): Migrate data persistence to PostgreSQL
         using Railway's PostgreSQL plugin. Create proper tables, a
         db.js connection module, and update all read/write operations.
       Option B (temporary): Add a clear ⚠️ WARNING comment at the top
         of every file-based read/write function, AND add a startup
         check that automatically re-seeds the default admin user if
         users.json is missing or empty.
   • State which option you chose and why.

2.7 — JWT_SECRET enforcement
   • Remove any hardcoded fallback JWT secret string
   • At startup, check for process.env.JWT_SECRET
   • If it is NOT set, throw an error and halt:
       if (!process.env.JWT_SECRET) {
         throw new Error("FATAL: JWT_SECRET environment variable is not set.");
       }
